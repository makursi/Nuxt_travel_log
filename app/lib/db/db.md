| 模式                                                  | 架构方式                                                               | 你的代码怎么连？                                                                    | 适用场景                                                                    |
| :---------------------------------------------------- | :--------------------------------------------------------------------- | :---------------------------------------------------------------------------------- | :-------------------------------------------------------------------------- |
| A. 独立服务模式<br>(你之前问的 `@libsql/server`)      | 客户端 - 服务器 (C/S)<br>数据库作为一个独立的 HTTP 程序运行在后台。    | 代码通过 HTTP 请求 (`http://localhost:8080`) 连接。<br>就像访问一个 API 接口。      | - 前端直接连库（不推荐但可行）<br>- 模拟云端 Turso 环境<br>- 多语言混合开发 |
| B. 嵌入式模式<br>(你现在的 `@tursodatabase/database`) | 进程内调用 (In-Process)<br>数据库作为你 Node.js 程序的一部分直接运行。 | 代码通过 函数调用 直接读取文件 (`file:local.db`)。<br>就像读取一个 txt 文件一样快。 | - Nuxt/Node.js 后端开发 (推荐)<br>- 高性能本地开发<br>- 单语言全栈          |

# 模式对比

| 模式                                     | 命令                   | 适用场景           | 工作原理                                                                                                                                     |
| :--------------------------------------- | :--------------------- | :----------------- | :------------------------------------------------------------------------------------------------------------------------------------------- |
| 迁移模式 (Migrations) ✅ (推荐/生产环境) | `generate` + `migrate` | 正式开发、生产环境 | 1. 对比代码生成 SQL 文件 (`generate`)<br>2. 手动审查 SQL 文件<br>3. 执行 SQL 文件应用到数据库 (`migrate`) <br>优点：可控、可版本管理、安全。 |
| 推送模式 (Push) ⚠️ (仅原型开发)          | `push`                 | 快速原型、临时测试 | 直接对比代码和数据库，自动修改数据库结构。<br>缺点：没有 SQL 文件记录，容易误删数据，不适合团队协作。                                        |

# Better Auth --- verification schema 变量名和字段名发生冲突

verification 表定义完全符合 Better Auth 的要求：
变量名：expiresAt, createdAt, updatedAt (驼峰命名 ✅)
数据库列名："expires_at", "created_at", "updated_at" (蛇形命名 ✅)
